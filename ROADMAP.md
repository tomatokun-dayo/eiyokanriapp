# 離乳食栄養管理アプリ ロードマップ

最終更新: 2026-07-02

## 現在の状態

静的HTML/CSS/JSのプロトタイプ。モックから実用最小限へ移行中。

実装済み:

- 1食あたりの食材入力（行は可変。＋追加 / ×削除、初期3行）
- 入力単位は g / 小さじ / 大さじ（換算は共通値: 小さじ5g・大さじ15g）
- 食材マスタは外部ファイル化済み（10品、`state` / `spoonGrams` / `source` 付き）
  - 100gあたり栄養値は文科省 食品成分DBと照合済み（食品番号・参照URL保持）
- 月齢別ラインは外部ファイル化済み（値は仮のまま）
- localStorage 保存（`eiyokanri.entries.v1`）
  - 今日ビュー（合計・ログ・器・件数）は当日分のみ表示
  - リセットは当日分のみ削除
  - 7日間グラフは保存済み実データを日別集計
- 不足栄養素の提示（目安の70%未満を最大3つ、補う食材候補つき）
  - 対象: エネルギー / たんぱく質 / 鉄 / カルシウム
- 食材の状態管理（未導入 / 導入済み / 注意中 / 避ける）
  - 食材データ一覧のセレクトで変更、`eiyokanri.foodStates.v1` に永続化
  - 「未導入」「避ける」は提案候補から自動除外

主なファイル:

- `index.html`
- `styles.css`
- `app.js`（UI・ロジック一体。約1,100行）
- `data/food-master.js`（`window.FOOD_MASTER`）
- `data/age-targets.js`（`window.AGE_TARGETS`）
- `README.md`

## 設計方針（明文化）

- **実行形態**: ビルドツールなし・ESモジュール不使用・`file://` 直開きで動くことを維持する。外部化は `<script defer>` の読み込み順で行う
- **データ**: localStorage キーはバージョン付き（`eiyokanri.*.v1`）。スキーマ変更時は移行関数を書いてキーを v2 に上げる
- **記録の意味**: entry は「摂取グラム」だけを持ち、栄養値は表示時に食材マスタから計算する。マスタの栄養値を修正すると過去の表示にも遡及する（意図した仕様）
- **DOM生成**: 新規・改修部分は `createElement` + `textContent`（innerHTML は使わない。セキュリティフック方針）
- **ロジック分離**: 計算・提案・換算などの純関数は将来 `logic/` 系ファイルに切り出し、Node で単体検証できる形を目指す
- **位置づけ**: 医療・栄養指導の代替ではなく家庭内の目安。UI上にも免責の一文を置く

## 既知の課題

- `todayKey` がページ読み込み時に固定される（開きっぱなしで日付をまたぐとズレる）
- `spoonGrams` はデータ構造のみで換算に未結線（全食材が共通値）
- `AGE_TARGETS` は仮値。「1日総摂取か・離乳食から取る分か」「母乳・ミルクの扱い」が未決定
- localStorage はオリジン単位。**http(s) 配信へ移行すると `file://` 時代の記録は見えなくなる** → エクスポート機能を先に用意すること
- git 管理がない（変更履歴・巻き戻し手段がない）
- 自動テストがない（都度の Node アドホック検証のみ）

## ロードマップ（優先順）

### 0. git 管理の導入 — 最優先・最小コスト

- `git init` → 現状を初回コミット
- 以後は作業単位でコミット（機能ごと）
- 理由: ここまでの資産に履歴と巻き戻し手段がない状態を解消する

### 1. スプーン換算の結線（小タスク）

- `amountToGrams()` を「食材の `spoonGrams` を優先、未設定は共通値へフォールバック」に拡張
- 単位切替時の相互換算・ログの「約◯g」表示も食材別値に追従させる
- あわせて、がゆ・ペースト等の実態に合う換算値をデータ側で精査

### 2. データのエクスポート / インポート

- 全記録＋食材状態を JSON で書き出し / 読み込み（ファイルダウンロード・選択）
- 目的: バックアップ、機種変更・ブラウザ変更への備え、**将来のホスティング移行（オリジン変更）前の必須通路**
- CSV 書き出しは集計用途として後続で検討

### 3. 日付またぎ対応

- 「今日」の判定を描画時に計算する（または日付変化を検知して再描画）
- 開きっぱなしでも日付が変われば今日ビューが切り替わること

### 4. 月齢別ラインの正式化

- 先に仕様を決める（コードより判断が先）:
  - 「1日の総摂取目安」か「離乳食から取りたい目安」か
  - 母乳・ミルク分の扱い
  - 12-18か月区分を追加するか
- 決定後、食事摂取基準に基づき `AGE_TARGETS` を差し替え、出典を持たせる
- UI の「仮」表記を出典付き表記に変更

### 5. 経験メモ（安全・経験管理の残り）

- 食材ごとに「食べられた / 嫌がった / 体調変化メモ」を残せるようにする
- 初回導入日を記録（記録追加時に未導入→導入済みへの自動昇格も検討）
- 保存は `eiyokanri.foodNotes.v1`（新キー）

### 6. 入力体験の改善 — 「皿が複数ある」問題の本命

- 前回の食事をコピー
- よく使う献立セット（皿テンプレート）の登録・ワンタップ呼び出し
- 食材ごとによく使う単位・量の記憶
- 食材検索
- 行可変化（実装済み）はここへの応急対応。テンプレートが入れば入力コストが大きく下がる

### 7. app.js の分割リファクタ

- UI 大改修（項目8）の前にやる
- 案: `logic/nutrition.js`（集計・目安）、`logic/suggestions.js`（提案）、`logic/units.js`（換算）、`store.js`（localStorage）、`app.js`（UI・イベント）
- ESモジュールは使わず `<script defer>` の順序読み込みで分割
- 純関数化した部分に Node の簡易テストを付ける

### 8. UIブラッシュアップ（モバイル前提）

- 1日の状況が一目で分かる要約
- グラフは必要なときだけ展開
- 不安を煽らない表現の総点検＋免責の一文を表示
- タップ領域・情報密度の整理

### 9. 将来: ホスティング / PWA 化

- 前提: 項目2（エクスポート/インポート）完了後
- 手順: 移行前に全データをエクスポート → 新オリジンでインポート
- クラウド同期・家族共有はさらにその後（認証・個人情報の扱いを設計してから）
